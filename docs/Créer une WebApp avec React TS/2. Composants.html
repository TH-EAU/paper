Les Composants en React

Résumé du cours complet : React propose un système de composants sous forme de fonctions JS. Celles-ci renvoient du JSX, langage déclaratif permettant d’intégrer du JS dynamique directement dans le HTML. Les composants ont un cycle de vie, qui leur permet de mettre à jour leur state ou état. Chaque changement d’état déclenche la mise à jour du composant, et donc de son rendu.

Le problème : performances, répétitions, et séparation du HTML et du JS

Considérons le cas suivant : Nous souhaitons créer un bouton qui va changer dynamiquement de texte lorsque l’on clique dessus. En JS Vanilla, nous aurions déclenché un évènement en important un script JS directement dans notre HTML, afin de modifier le DOM de notre page web :


index.html<button id="myButton">👉 Click me !</button>
<script src="scripts.js"></script>
script.js
button.addEventListener("click", () => {
button.disabled = true;
button.textContent = 'Loading';
})

Ici vous pouvez visualiser par la pratique le déclenchement d’un évènement en JS 👆

Il s’agit de l’approche impérative. Le premier point de frustration de cette approche est que la modification du DOM est très couteuse en performance. Pour de petites modifications elle peux être intéressante, mais lorsque l’on créé une application entièrement basée sur des interactions dynamiques, ou des échanges de données, cela devient vite un enfer.

Un autre point de frustration vient du fait que le code HTML que nous déclarons n’est pas réutilisable. En d’autre termes, si je voulais dupliquer le bouton que je viens de créer, je vais devoir le réécrire.

[!quote] Réécrire un bouton ? Ca va très vite !

Effectivement, mais si notre bouton avait un comportement complexe, ou bien qu’il affiche une liste de données au moment ou l’on clique dessus. La liste qui suis le bouton devras elle aussi être réécrite. Pour peu que vous utilisiez ce principe partout dans votre application, vous allez réécrire toutes vos balises autant de fois.

Pour finir avec cette approche, il n’y pas de lien réel entre votre HTML et votre JS. La fonction que vous allez déclencher ira modifier directement le HTML dans votre DOM, mais vous n’avez pas le contrôle de ce qui est affiché.

Le seul moyen de vérifier que vous affichez bien les bonnes données à l’écran, c’est soi de regarder avec vos yeux (oui ça parait logique), soit de de nouveau manipuler le DOM en allant chercher l’information au bon endroit. Et pour savoir ou est l’information, bienvenu dans l’enfer des id en HTML.

En résumé, l’idée est que l’approche de base peut paraitre simple et efficace lorsque nous traitons d’un cas isolé, mais lorsque que nous dézoomons pour regarder à l’échelle d’une application, cette solution devient trop lourde et demande en réalité une extrême rigueur pour obtenir un produit viable.

C’est pourquoi les équipes de développement de Facebook ont souhaité changer de paradigme de programmation pour les interfaces utilisateurs, en nous proposant l’approche déclarative.

La solution : L’approche déclarative, ou le JSX/TSX , et React

Le JSX/TSX est un langage de programmation extrêmement proche du HTML. Il permet en revanche d’intégrer du javascript directement dans le HTML. C’est plus simple non ?

En terme de performances, React propose un système de ShadowDOM. Nous y reviendrons, mais il s’agit d’un système qui “imite” le DOM de votre page web, mais qui est beaucoup plus accessible par le JS. Le résultat sur votre page web est le même, mais vous n’avez jamais besoin de modifier directement votre DOM, c’est donc plus rapide.

Le fait que le JS soit permis dans le JSX/TSX, nous permet d’écrire directement notre code lié aux interactions et aux données, au même endroit que notre code HTML. Donc dans le même fichier, j’aurais :

  • Du code JS
  • Du HTML augmenté avec du JS dedans

A quoi ça peut nous servir de mettre du JS dans du HTML ?

A ce stade, nous pouvons faire le parallèle avec une autre technologie, le PHP. Celui-ci vous propose d’intégrer des données dans votre html, comme le ferait le JSX, entre des balises <?php ?>. Alors si le PHP le fait très bien pourquoi utiliser React ?

La principale différence entre PHP et React est que PHP est un langage serveur et donc ne peux pas être exécuté chez le client. Les données inscrites par PHP ne sont pas dynamiques, il faut recharger la page pour les voir évoluer, ce qui n’est pas le cas de React

Composant

Plus haut, j’évoque le fait que tout notre code sera au même endroit. Mais vous savez qu’il est important de bien séparer son code. Et c’est la que React prend tout son sens avec la notion de composant.

Les composants sont des fonctions JS, qui peuvent êtres appelées partout dans votre application sous forme de balise HTML.

[!tip] A l’instar de la programmation objet ou tout les fichiers sont des classes, en React la quasi-totalité des fichiers sont des composants. Voyez les comme des briques que vous pouvez réutiliser à l’infini dans vos projets pour gagner un temps monstre sur la réalisation de votre interface. C’est presque de la triche !
